# ADR-002: Использование WebSocket STOMP для real-time взаимодействия

## Контекст

Приложению требуется real-time взаимодействие между организатором квиза и участниками.

Во время активной quiz-сессии система должна поддерживать:

- одновременный показ текущего вопроса всем участникам
- возможность отвечать только пока вопрос активен
- рассылку изменений таймера и состояния сессии
- обновление лидерборда
- уведомления о старте квиза, переходе к следующему вопросу и завершении квиза

Рассматривались несколько вариантов:

1. HTTP polling
2. Server-Sent Events
3. Raw WebSocket
4. WebSocket с STOMP

### HTTP polling

Polling прост в реализации, но плохо подходит для real-time квиза. Клиенты должны постоянно опрашивать сервер, что увеличивает задержку и создаёт лишнюю нагрузку.

### Server-Sent Events

Server-Sent Events хорошо подходят для односторонней передачи событий от сервера к клиенту. Но в квизе нужна двусторонняя коммуникация:

- сервер отправляет клиентам изменения вопроса и состояния сессии
- клиенты отправляют ответы и события участия обратно на сервер

SSE можно было бы сочетать с обычными HTTP-запросами для действий клиента, но тогда real-time логика была бы разделена между разными способами коммуникации.

### Raw WebSocket

Raw WebSocket поддерживает двустороннюю коммуникацию и подходит для квиза. Но при таком подходе пришлось бы самостоятельно проектировать маршрутизацию сообщений, формат сообщений, подписки и обработку разных типов событий.

### WebSocket с STOMP

STOMP поверх WebSocket даёт более структурированную модель обмена сообщениями: destinations, topics и application endpoints. Он хорошо интегрируется со Spring Boot и подходит под модель quiz-сессий.

Примеры destinations:

```text
/app/sessions/{code}/answer
/app/sessions/{code}/start
/app/sessions/{code}/next-question

/topic/sessions/{code}/state
/topic/sessions/{code}/question
/topic/sessions/{code}/leaderboard
